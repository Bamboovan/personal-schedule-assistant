# 个人日程助手智能体 - 项目总结与复试问答

## 📋 项目总结

### 一、项目背景

这是我学习智能体开发的入门项目，基于 NexAU 框架实现了一个个人日程管理助手。选择这个项目的初衷是因为自己经常需要管理课程、作业和考试安排，希望能有一个智能工具帮助我从聊天记录和笔记中自动提取日程信息。

### 二、完成的工作

#### 1. 环境搭建与框架学习
- 学习并使用了 NexAU v0.3.9 智能体框架
- 配置了多模型支持（Qwen/DeepSeek/硅基流动）
- 搭建了完整的项目结构

#### 2. 工具层设计与实现
设计了 4 个核心工具：
- **add_event** - 添加日程
- **modify_event** - 修改日程
- **query_event** - 查询日程
- **delete_event** - 删除日程

每个工具都包含 YAML 描述层和 Python 实现层，使用 dataclass 定义数据结构，JSON 文件持久化存储。

#### 3. 文档解析器实现
实现了 ScheduleParser 类，能够：
- 识别多种时间格式（标准日期、相对时间、月日格式）
- 从 Markdown 文件和聊天记录中提取日程信息
- 支持关键词匹配（会议、安排、约会等）

#### 4. 智能体集成
- 编写系统提示词定义角色和工作流程
- 配置 config.yaml 整合所有组件
- 实现交互式命令行界面

#### 5. 测试与文档
- 编写单元测试覆盖核心功能
- 撰写用户指南和 README 文档

### 三、技术亮点

1. **分层架构设计**：工具描述（YAML）与实现（Python）分离，便于维护
2. **自然语言时间理解**：支持"明天下午 3 点"等相对时间表达
3. **持久化存储**：JSON 文件存储日程数据，支持数据持久化
4. **可扩展性**：模块化设计，易于添加新功能

### 四、遇到的挑战与解决方案

| 问题 | 解决方案 |
|------|----------|
| 正则表达式匹配中文失败 | 将 `\s*` 改为 `*` 解决中文空格匹配问题 |
| 相对时间转换复杂 | 建立相对时间映射表，结合 datetime 计算目标日期 |
| NexAU 框架文档较少 | 阅读源码和 examples 目录学习用法 |

### 五、收获与反思

**收获：**
- 掌握了智能体框架的基本使用方法
- 学习了工具调用（Tool Calling）的设计模式
- 提升了正则表达式和文本处理能力

**不足与改进方向：**
- 时间解析还不够完善，不支持"下周"、"月底"等模糊表达

### Q2: 你的时间解析模块是如何工作的？

**A:** 时间解析主要分三步：

1. **模式匹配**：使用正则表达式识别时间信息，支持 5 种模式：
   - 完整日期范围：`2026-03-10 15:00~16:00`
   - 完整日期 + 时间：`2026-03-10 15:00`
   - 月日格式：`3 月 10 日 15:00`
   - 相对时间范围：`明天 15:00~16:00`
   - 相对时间 + 时间：`明天 15:00`

2. **信息提取**：从匹配结果中提取日期和时间组分

3. **标准化**：将相对时间转换为标准格式 `YYYY-MM-DD HH:MM`
   ```python
   # 相对时间映射
   relative_days = {
       '今天': 0,
       '明天': 1,
       '后天': 2,
       '下周一': (7 - today.weekday()) % 7 + 7,
       # ...
   }
   ```

### Q7: 你在项目中遇到的最大困难是什么？如何解决的？

**A:** 最大的困难是**正则表达式匹配中文文本**的问题。

**问题描述**：我最初写的正则表达式 `r'(明天)\s*(\d{1,2}:\d{2})'` 无法匹配"明天 15:00"，但单独测试"明天"和"15:00"都能匹配。

**排查过程**：
1. 检查文本编码，确认是 UTF-8
2. 打印每个字符的 Unicode 码点，发现"明天"和"15:00"之间确实有一个普通空格 (0x20)
3. 尝试将 `\s*` 改为 ` *`（匹配普通空格）

**解决方案**：最终发现是正则表达式中使用了 `\s*` 但实际应该用 `*` 来匹配任意字符（包括无字符）。修改后问题解决。

这个经历让我学到了：调试正则表达式时要耐心，逐层排查问题。

### Q9: 解释一下你的 EventManager 类的设计思路

**A:** EventManager 类采用**简单直观**的设计原则：

**核心设计**：
1. **数据持久化**：使用 JSON 文件存储，初始化时加载，修改后保存
2. **内存缓存**：数据加载到 `self.events` 字典，避免频繁 IO
3. **CRUD 操作**：实现标准的增删改查接口

**代码结构**：
```python
class EventManager:
    def __init__(self):  # 初始化，加载数据
    def _load_events(self):  # 私有方法，加载 JSON
    def _save_events(self):  # 私有方法，保存 JSON
    def add_event(self):     # 添加
    def modify_event(self):  # 修改
    def query_events(self):  # 查询
    def delete_event(self):  # 删除
```

**设计权衡**：
- 选择 JSON 而非数据库：简单、易调试、适合小规模数据
- 使用字典存储：O(1) 查找效率，适合按 ID 查询

### Q10: 如果用户说"把明天的会议取消"，你的系统如何理解并执行？

**A:** 这个请求的处理流程是：

1. **LLM 理解意图**：用户要删除日程
2. **LLM 提取信息**：时间是"明天"，事件类型是"会议"
3. **工具调用**：
   - 先调用 `query_event` 查询明天的所有会议
   - 如果有多个会议，需要用户确认具体哪个
   - 获取 event_id 后调用 `delete_event`

**关键代码**（由 LLM 自动生成）：
```python
# 第一步：查询
events = query_event(start_date="2026-03-10", end_date="2026-03-10", keyword="会议")

# 第二步：删除（假设只有一个会议）
delete_event(event_id=events["events"][0]["event_id"])
```

这个场景展示了智能体如何**组合使用多个工具**完成复杂任务。
